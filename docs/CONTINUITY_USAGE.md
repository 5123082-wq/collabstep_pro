# Инструкция По Использованию CONTINUITY.md

## Что Это Такое

`CONTINUITY.md` — это compact active-state ledger для текущей работы. Он нужен для handoff между агентами и для продолжения работы после длинных чатов или context compaction.

## Как Это Работает Сейчас

1. Правило в `.cursor/rules/continuity.mdc` применяется автоматически.
2. По умолчанию агент читает только часть `CONTINUITY.md` выше `<!-- STOP LINE: ACTIVE CONTEXT ENDS -->`.
3. Архив continuity в `docs/archive/continuity/` не читается по умолчанию и открывается только если нужен исторический контекст.
4. В `CONTINUITY.md` хранятся только факты, active context, open threads и handoff-состояние. Не планы релизов и не длинная история проекта.

## Что Не Нужно Хранить В CONTINUITY.md

Не складывайте туда:
- полный roadmap;
- release history;
- стенограммы разговоров;
- секреты и приватные URL;
- длинные исторические журналы.

Для этого используются:
- `docs/ROADMAP.md` — планы и статусы реализации;
- `docs/platform/changelog.md` — релизная история;
- `docs/archive/continuity/` — исторические снимки continuity.

## Когда Агент Должен Обновлять CONTINUITY.md

- при значимой новой фиче или кросс-секционном изменении;
- при паузе в незавершённой работе;
- при завершении крупного этапа, если это важно для handoff;
- при изменении ключевых решений или open threads.

## Когда Пользователь Может Явно Попросить Обновление

### Если агент потерял контекст

```text
Используй @CONTINUITY.md и продолжай.
```

### Если началась новая крупная ветка работы

```text
Обнови CONTINUITY.md под новую активную задачу и зафиксируй open threads.
```

### Если работа остановилась на полпути

```text
Синхронизируй CONTINUITY.md: зафиксируй, где остановились, что уже сделано и что остаётся.
```

## Что Обычно Находится В Active Section

Выше `STOP LINE` обычно находятся:
- краткий active platform context;
- открытые хвосты и follow-up задачи;
- рабочие правила для чтения файла;
- ссылки на docs, которые нужно проверять по умолчанию.

## Рекомендации

1. Держите активную часть короткой и factual.
2. Если active section разрастается, переносите старую деталь в `docs/archive/continuity/`.
3. Если задача про roadmap или релиз, сначала смотрите `docs/ROADMAP.md` и `docs/platform/changelog.md`, а не пытайтесь вытянуть это из `CONTINUITY.md`.
4. Если факт не подтверждён, помечайте его как `UNCONFIRMED`.
