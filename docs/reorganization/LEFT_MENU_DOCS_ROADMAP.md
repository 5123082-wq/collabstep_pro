# Роадмап: выравнивание пользовательской документации по левому меню

**Статус:** draft  
**Владелец:** product/engineering  
**Создан:** 2026-01-07  
**Последнее обновление:** 2026-01-07  
**Источник структуры меню:** `apps/web/components/app/LeftMenu.config.ts`

## Цель

Навести порядок в пользовательской документации так, чтобы:
- верхний уровень документации повторял разделы левого меню,
- внутри разделов документы были сгруппированы логично (не 1:1),
- были описаны верхнее меню, правое меню и кнопка "Создать",
- одинаковые названия страниц различались префиксом раздела.

## Ограничения и правила

- Базовая структура — `docs/modules/`; переносим и переименовываем существующие документы.
- Документация одна на все роли, но различия в доступах и поведении обязательно фиксировать.
- Разделы под фичефлагами: фиксируем структуру и помечаем planned в индексах.
- Кнопка "Создать": общий принцип описать отдельно, затем отдельный подраздел в каждом разделе.
- Правое меню: один общий раздел.
- Верхнее меню: часть общего раздела навигации.
- Заголовки H1 с префиксом раздела (пример: `Маркетплейс — Каталог шаблонов`).
- Язык: RU, EN только при необходимости терминов или UI-лейблов.
- Скриншоты обязательны для ключевых экранов.

## Роадмап (3 этапа)

### Этап 1 — структура и каркас

**Задачи:**
- Снять актуальный список разделов и страниц из `LeftMenu.config.ts`.
- Собрать карту "раздел меню → документы" (где уже есть, где нужны новые).
- Сформировать структуру по левому меню на базе существующих документов (перенос/переименование).
- Для фичефлагов зафиксировать planned в индексах без создания новых файлов.
- Зафиксировать стандарт заголовков с префиксом раздела.

**Результат:**
- Есть каркас документов по всем пунктам левого меню.
- Все документы имеют уникальные H1 через префикс раздела.
- Подготовлена таблица соответствий "UI раздел ↔ документация".

### Этап 2 — навигация и поведение

**Задачи:**
- Создать единый документ "Навигация приложения" (левое/верхнее/правое меню).
- Добавить раздел про кнопку "Создать": общий принцип + в каждом разделе отдельный подраздел.
- Для каждого раздела описать ключевые сценарии и различия ролей.
- Привязать каждый пункт меню к соответствующим документам (ссылки).

**Результат:**
- Появился документ навигации с описанием верхнего и правого меню.
- Во всех разделах описано поведение кнопки "Создать".
- В каждом разделе обозначены различия ролей и доступность фич.

### Этап 3 — визуал и контроль качества

**Задачи:**
- Сделать скриншоты ключевых экранов (по одному на раздел + навигация).
- Проверить уникальность названий и отсутствие дубликатов в H1.
- Проверить, что порядок в индексах соответствует левому меню.
- Мини-проход по ссылкам и структуре (без переписывания контента).

**Результат:**
- Документация визуально подтверждена скриншотами.
- Навигация и порядок соответствуют левому меню.
- Дубликаты названий устранены через префиксы.

## Рекомендованный стандарт названий

- H1: `{Раздел} — {Страница}`  
  Пример: `Проекты и задачи — Архив`
- Файл: `{section}-{page}.md` (kebab-case)  
  Пример: `projects-archive.md`

## Примечания для исполнителя

- Не пытаться "переписать все сразу": сначала каркас, потом контент.
- Структуру меню брать только из `LeftMenu.config.ts`.
- Скриншоты складывать в один каталог (например `docs/assets/screenshots/`).
