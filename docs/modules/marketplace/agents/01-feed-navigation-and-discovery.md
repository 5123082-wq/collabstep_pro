# Subagent 01 — IA, навигация и discovery-first feed

**Статус:** completed  
**Владелец:** product/frontend  
**Последнее обновление:** 2026-03-09

## Миссия

Перестроить первый слой модуля из `shop-first Marketplace` в `discovery-first Каталог`.

## Зависимости

- `docs/modules/marketplace/marketplace-implementation-plan.md`
- `docs/modules/marketplace/marketplace-overview.md`

## Scope

- пользовательское переименование раздела в `Каталог`;
- новая IA первого слоя;
- концепт главной discovery-ленты;
- карточка решения;
- подборки, сохранённое, переход к автору;
- понижение приоритета корзины/заказов/продаж в навигации.

## Out of Scope

- данные автора;
- логика publish-flow;
- сделки и платежи;
- импорт решения в PM.

## Пошаговая работа

1. Зафиксировать current IA и target IA.
2. Согласовать новые user-facing лейблы раздела и подпунктов.
3. Описать главную страницу `/market` как визуальную ленту.
4. Описать карточку решения:
   - hero/preview;
   - тип сущности;
   - автор;
   - теги;
   - detail surface для действий вместо CTA на самой плитке.
5. Описать, какие разделы остаются во втором слое.
6. Сверить, что шаблоны, готовые решения и услуги остаются в одном модуле.

## Артефакты

- обновлённый overview;
- обновлённые docs по шаблонам/решениям/услугам/сохранённому/подборкам;
- список новых user-facing названий для навигации.

## DoD

- модуль не ощущается как магазин;
- IA ведёт к discovery, а не к checkout;
- карточка решения всегда показывает автора;
- reused-механика `В проект` сохранена как главный CTA.

- отметить, какие лейблы пока меняются только в документации;

## Результат C1 (2026-03-09)

### Что внедрено

- `/market` больше не редиректит в шаблоны и работает как главная discovery-лента каталога;
- в UI обновлены user-facing лейблы первого слоя и вторичного слоя;
- после corrective task discovery-плитки шаблонов, готовых решений и услуг показывают только название, краткое описание, хэштеги, автора и demo-метрики;
- действия `Открыть`, `Сохранить`, `В проект`, `Запросить адаптацию` вынесены из плиток в detail surface;
- страницы `Готовые решения`, `Услуги` и `Подборки` перестали быть чистыми shop-first заглушками и получили discovery-first framing;
- корзина, сделки и доступ, публикация и кабинет автора явно переосмыслены как secondary / author-facing surfaces.

### Лейблы, которые реально изменены в UI

- `Маркетплейс` -> `Каталог`
- `Каталог шаблонов` -> `Шаблоны`
- `Готовые проекты` -> `Готовые решения`
- `Пакеты услуг` -> `Услуги`
- `Категории и подборки` -> `Подборки`
- `Избранное` -> `Сохранённое`
- `Мои заказы` -> `Сделки и доступ`
- `Кабинет продавца` -> `Мои публикации`
- `Публикация в маркетплейсе` -> `Опубликовать в каталог`
- `Корзина` -> `Корзина и оформление`

### Лейблы / framing, которые пока закреплены только в документации или частично в UI

- `discovery-first` / `secondary surface` — это продуктовые термины для IA, а не отдельные UI labels;
- полноценное позиционирование `/p/:handle` как author-page каталога будет доведено в C2;
- `В проект` и service actions остаются в detail surface, но deep apply/import behavior всё ещё задача C4.
- analytics/permissions framing под новую IA не менялись в C1 и остаются задачей C5.

### Open Questions для C2

- как канонически маппить автора каталога на существующий `performer_profile`, если handle в mock/discovery данных не совпадает с живым профилем;
- какие блоки страницы автора обязательны в первом экране и что можно оставить только для expanded layout;
- нужно ли скрывать автора/карточку полностью, если публичный профиль есть, но author-page неполна.

### Open Questions для C3

- как пользователь управляет показом конкретной публикации на своей author-page;
- нужны ли разные дефолтные CTA для шаблона, готового решения и услуги внутри publish-flow;
- как кабинет `Мои публикации` должен показывать границу между PM-проектом и public listing без дублирования сущностей.
