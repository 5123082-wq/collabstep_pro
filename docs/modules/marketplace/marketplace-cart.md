# Каталог — Корзина и оформление

**Статус:** partial  
**Владелец:** product/engineering  
**Создан:** 2026-01-07  
**Последнее обновление:** 2026-03-09

## Назначение

Корзина и оформление остаются в модуле, но становятся secondary surface.

Они нужны для тех сценариев, где действительно есть cart/checkout behavior:

- платный шаблон;
- фиксированный оффер;
- покупка доступа к материалам.

## Навигация

- `/market/cart` — корзина и оформление

## Продуктовая роль

Этот раздел **не должен** определять первое впечатление о каталоге.

Правильная иерархия:

- сначала discovery;
- потом shortlist / author / reuse;
- только затем оформление, если оно вообще нужно.

## Текущее состояние

- user-facing label `Корзина и оформление` уже внедрён в UI;
- корзина живёт локально в persisted Zustand;
- расчёты работают только для `paid`;
- reuse-flow не ведёт сюда по умолчанию:
  - template detail сначала предлагает `Использовать в проекте`;
  - сервисы не попадают в cart-flow;
- checkout — демо-кнопка без реальной оплаты.

## Основные сценарии

1. Добавить платный шаблон в корзину.
2. Проверить состав и сумму.
3. Перейти к оформлению.
4. После оформления получить доступ.

## Ограничения и планируемое

- корзина не должна быть главным пунктом верхнего уровня в target IA;
- услуги не обязаны всегда использовать cart-flow;
- доступ и материалы после оформления должны быть согласованы с `Сделки и доступ`;
- C5 зафиксировал отсутствие реального checkout telemetry, payment session и access issuance как отдельный future scope.
