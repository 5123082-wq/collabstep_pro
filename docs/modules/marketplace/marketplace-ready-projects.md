# Каталог — Готовые решения

**Статус:** partial  
**Владелец:** product/engineering  
**Создан:** 2026-01-07  
**Последнее обновление:** 2026-03-09

## Назначение

`Готовые решения` — это публичные публикации проектов, созданных в PM и вынесенных в каталог как reusable starting point.

Это не просто объект покупки, а одна из форм ответа на вопрос:

`С чего мне начать проект быстрее?`

## Навигация

- `/market/projects` — готовые решения

## Источник данных

Источником является publish-flow из PM:

- проект создаётся и живёт в PM;
- затем пользователь создаёт публичную публикацию;
- только публикация попадает в каталог.

## Основные сценарии

1. Найти готовое решение в каталоге.
2. Открыть detail surface и посмотреть состав/позиционирование.
3. Перейти к автору решения.
4. Использовать решение как основу нового проекта.
5. Запросить адаптацию или внедрение, если решение не подходит как есть.

## Действия

- discovery-плитка не показывает CTA;
- главные действия живут в detail surface: `Использовать в проекте`, `Запросить адаптацию`, `Открыть автора`;
- checkout-слой не выведен в первый экран и не доминирует в UI.

## Статусы публикации

Сохраняются текущие состояния:

- `draft`
- `published`
- `rejected`

Новые статусы не вводятся без отдельной необходимости.

## Текущее состояние

- `/market/projects` уже существует как discovery-first витрина на mock-card данных;
- discovery-плитка ready solution показывает только название, краткое описание, хэштеги, автора и demo-метрики;
- действия вынесены в detail surface;
- `Использовать в проекте` открывает reuse modal:
  - по умолчанию создаёт новый PM-проект;
  - альтернативно импортирует решение в существующий проект как отдельный block задач;
- в ветке `создать новый проект` выбранная организация теперь становится каноническим PM context:
  - personal org -> personal project;
  - team org -> проект создаётся в org-bound workspace и получает минимальный team access bridge через `project_members`;
- apply/import не создаёт новую публикацию и не смешивает public listing с PM-project state;
- API публикации проекта существует;
- статус публикации уже отображается в PM-карточке проекта.

## Ограничения и планируемое

- витрина пока не построена поверх реальных публикаций;
- author-page связь в UI уже обозначена, но canonical author identity доводится в C2;
- требуется отдельная спецификация того, какие project-данные можно показывать публично;
- full real-publications feed остаётся future scope после C5;
- analytics/access gaps задокументированы в C5 как отдельный rollout backlog, без переоткрытия contracts C3-C4.
