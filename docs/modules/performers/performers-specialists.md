# Исполнители — Специалисты

**Статус:** partial
**Владелец:** product/engineering
**Создан:** 2026-01-07
**Последнее обновление:** 2026-03-15

## Назначение

Раздел «Специалисты» сейчас является публичным каталогом исполнителей, а в целевом контракте должен стать первой поверхностью картотеки людей: поиска, фильтрации, просмотра карточек контактов и перехода в контактный workflow.

## Навигация

- `/performers/specialists` — каталог специалистов
- `/p/:handle` — публичная визитка

## Текущее состояние (по UI и коду)

- Каталог использует публичные профили из `performer_profile` и отображает рейтинги.
- Поиск, фильтры и сортировки в каталоге специалистов сейчас в основном работают на клиенте; есть быстрые фильтры и блок «Все фильтры».
- Карточка специалиста поддерживает действия: «Пригласить», «Запросить интервью», «Открыть визитку».
- Публичная визитка отображает реальные данные профиля, портфолио, кейсы и отзывы.
- В PM параллельно существует отдельный people picker `/api/pm/projects/[id]/member-candidates`, который ищет зарегистрированных пользователей платформы по имени, email и должности и нужен для быстрого operational add в проект.

## Модель данных (кратко)

Профиль исполнителя (`performer_profile`):
- `specialization` — специализация/роль
- `skills` — массив навыков
- `bio` — описание
- `rate` — ставка (число)
- `employmentType` — `fulltime` | `parttime` | `contract`
- `location`, `timezone`
- `isPublic` — видимость в каталоге

Схема карточки специалиста в каталоге строится из профиля и дополняется UI‑полями (рейтинг, языки, формат работы).

## Канонический механизм поиска людей

Поиск людей в платформе должен состоять из трёх взаимодополняющих слоёв.

1. **Публичный discovery-layer**  
   Каталог `/performers/specialists` ищет публичные карточки исполнителей и нужен для discovery, сравнения и первого контакта.

2. **Operational people picker внутри PM**  
   `/api/pm/projects/[id]/member-candidates` решает другую задачу: быстро найти уже зарегистрированного человека и понять его статус относительно проекта и организации.

3. **Приватный relation-layer организации**  
   Поверх каталога нужен приватный слой `контакты / shortlist / relation status`, чтобы организация видела не только публичные данные человека, но и историю общения, заметки, статус допуска и связь с проектами.

## Фильтры каталога и картотеки

UI:
- Быстрые фильтры: роль, формат работы, сортировка, поиск.
- «Все фильтры»: навыки, язык, ставка.

Поддерживаются параметры URL:
- `q` — поиск
- `role` — роль
- `skills` — список навыков через запятую
- `lang` — язык
- `format` — `remote` | `office` | `hybrid`
- `rateMin`, `rateMax` — фильтр ставки
- `sort` — `rating` | `cost` | `new`
- `page` — страница

Целевые фильтры для картотеки людей:
- категория: специалист / команда / подрядчик;
- специализация и навыки;
- язык, локация, часовой пояс;
- формат работы, занятость, доступность;
- ставка и диапазон стоимости;
- статус отношений с организацией: не общались / есть диалог / приглашён / preview / ждёт одобрения / в команде;
- привязка к организации и к конкретному проекту.

## Карточка контакта

Целевой контактный экран должен состоять из двух слоёв.

- **Публичный слой:** имя, handle, специализация, bio, навыки, портфолио, кейсы, рейтинг, условия работы.
- **Приватный слой организации:** заметки, теги, shortlist, связанные проекты, статус допуска, кто из команды уже общался с кандидатом.

Основные действия на карточке:
- `Написать`
- `Сохранить в контакты`
- `Добавить в shortlist`
- `Предложить preview проекта`
- `Пригласить в организацию`
- `Добавить в проект после одобрения`

## Основные сценарии

1. Найти специалиста через фильтры и поиск.
2. Открыть визитку специалиста.
3. Начать контакт с кандидатом и сохранить его в приватную картотеку организации.
4. Дать кандидату preview-доступ к проекту и провести его через approval flow.
5. Пригласить специалиста в организацию/проект.
6. Создать и опубликовать собственную карточку.

## Создание

Текущий путь:
1. Перейти в `/settings/performer` или `/onboarding/create-profile`.
2. Заполнить специализацию, навыки, bio, ставку и тип занятости.
3. При необходимости заполнить handle, языки, форматы работы, портфолио и кейсы.
4. Включить `isPublic`, чтобы карточка появилась в каталоге.
5. Сохранить профиль.

Целевой путь:
1. Пользователь открывает единый кабинет.
2. В разделе карточки исполнителя создаёт private draft через тот же `performer_profile`.
3. Заполняет базовые поля, затем портфолио и кейсы.
4. Публикует карточку в каталог отдельным осознанным действием.

## Спорные статусы и роли

- **`isPublic=false`** — профиль доступен только владельцу.
- Контакт и приглашение внешнего кандидата может создать только owner/admin организации.
- Быстрое direct add в PM-проект не заменяет discovery/contact flow для внешних кандидатов.

## Доступность по ролям

- Каталог доступен всем авторизованным пользователям.
- Редактирование профиля доступно только владельцу профиля.

## API (MVP)

- `GET /api/performers?specialization=&limit=&offset=` — список публичных профилей
- `GET /api/performers/:userId` — публичный профиль
- `GET /api/pm/projects/[id]/member-candidates?q=` — server-side people picker для PM-проекта
- `GET /api/me/performer-profile` — профиль текущего пользователя
- `POST /api/me/performer-profile` — создать/обновить профиль
- `PATCH /api/me/performer-profile/visibility` — изменить публичность
- `GET /api/me/performer-profile/portfolio` — список работ
- `POST /api/me/performer-profile/portfolio` — добавить работу
- `DELETE /api/me/performer-profile/portfolio?id=` — удалить работу
- `POST /api/me/performer-profile/portfolio/upload-url` — получить upload-url для файла
- `GET /api/me/performer-profile/cases` — список кейсов
- `POST /api/me/performer-profile/cases` — добавить кейс
- `DELETE /api/me/performer-profile/cases?id=` — удалить кейс
- `GET /api/performers/:userId/rating-summary` — сводный рейтинг
- `GET /api/performers/:userId/ratings` — список отзывов
- `POST /api/performers/:userId/ratings` — оставить рейтинг (требуется `projectId`)
- `POST /api/performers/:userId/invite-to-organization` — пригласить в организацию/проект

## Взаимодействия с другими разделами

- **Проекты и задачи:** people picker проекта отвечает за быстрый add известных пользователей; каталог и картотека отвечают за discovery и контактный workflow.
- **Организация:** инвайты в организацию.
- **Кабинет пользователя:** создание карточки исполнителя должно идти через единый cabinet shell, а не через разрозненные страницы.

## Ограничения и планируемое

- Фильтры каталога пока не являются каноническим server-side people search для всей платформы.
- Нет приватной картотеки контактов организации.
- Нет карточки контакта с заметками, тегами и shortlist.
- Нет единого UX-перехода `нашёл человека -> написал -> дал preview -> одобрил -> добавил в проект`.
- **Условия рейтинга:** рейтинг доступен только для участников проекта со статусом `completed` | `archived`, рейтить могут участники проекта, не `viewer`; обе стороны должны быть участниками проекта.
- **Портфолио (MVP/следующий этап):**
  - MVP: загрузка файлов через Vercel Blob, `file_url` хранит blob‑ссылку.
  - Следующий этап: подключение внешних источников/ссылок (Behance, Dribbble, Notion, Drive и т.д.).
